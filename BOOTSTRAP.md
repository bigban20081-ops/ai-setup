## Фаза 3: Генерация файла инструкций

### Прочитай базовые принципы

Прочитай файл `core/principles.md` из этого репозитория — он содержит универсальные принципы, которые войдут в итоговый файл.

### Структура итогового файла

Файл инструкций состоит из двух частей:

**Часть 1 — кастомная** (если файл уже существует — сохранить как есть):
```
[существующее содержимое пользователя]
```

**Часть 2 — управляемая ai-setup** (добавить в конец):
```
<!-- ai-setup:start -->
[сгенерированное содержимое]
<!-- ai-setup:end -->
```

**Правила работы с маркерами:**
- Если найдены оба маркера корректно — заменить только содержимое между ними.
- Если маркеры не найдены — добавить блок в конец файла.
- Если найден только один маркер или порядок нарушен — добавить новый блок в конец, сообщить пользователю о проблеме.

### Что генерировать внутри маркеров

Сгенерируй разделы на основе:
1. Универсальных принципов из `core/principles.md`
2. Стек-специфичных правил — сгенерируй сам на основе определённого стека

**Стек-специфичные правила** — примеры того, что генерировать:

*Django:*
- Паттерн ViewSet: `permission_classes`, фильтрация по owner, `queryset.none()` как дефолт
- Миграции: именование, порядок применения
- ORM: всегда фильтровать tenant-scoped модели по owner

*Next.js:*
- Server / Client component разделение
- Роутинг и metadata
- Работа с cookies и auth

*Docker:*
- Логирование только в stdout
- Не хардкодить порты

*Celery:*
- Идемпотентность задач
- Обработка ошибок и retry-логика

Адаптируй под реальный стек проекта. Используй свои знания о фреймворке — не копируй шаблонные правила вслепую.

### Формат разделов внутри маркеров

```markdown
## Принципы разработки [ai-setup]

### Workflow
[из core/principles.md — раздел Workflow]

### Git
[из core/principles.md — раздел Git]

### Безопасность
[из core/principles.md — раздел Безопасность]
[+ стек-специфичные правила]

### [Язык/Фреймворк]
[специфичные паттерны для данного стека]

### Качество кода
[из core/principles.md — раздел Принципы кода]

### Дизайн-система
[из core/principles.md — раздел Дизайн-система]

### SEO
[из core/principles.md — раздел SEO]

### Документация
[из core/principles.md — раздел Документация]
```

## Фаза 4: Структура памяти

Создай директорию `.agents/memory/` в корне проекта (если не существует).

Создай файл `.agents/memory/MEMORY.md`:

```markdown
# Memory — [Название проекта]

## Проект
- **Название**: [заполнить]
- **Стек**: [определённый стек]

## Файлы памяти
- **insights.md** — архитектурные наблюдения, грабли, неочевидные паттерны
```

Создай файл `.agents/memory/insights.md`:

```markdown
# Insights — [Название проекта]

Записывай сюда неочевидные паттерны, грабли и архитектурные решения,
которые важно помнить между сессиями.
```

Сообщи пользователю:
```
Создал структуру памяти в .agents/memory/
Записывай наблюдения в insights.md между сессиями.
```

## Фаза 6: Настройка MCP

Прочитай файл `mcp/core.json` из этого репозитория.

Для каждого MCP-сервера:
1. Показать название и описание
2. Спросить нужен ли он (некоторые опциональны)
3. Если нужен — запросить API-ключи (они не хранятся в репо)
4. Добавить в конфиг AI-инструмента

**Расположение конфигов MCP по инструментам:**

| Инструмент | macOS | Windows | Linux |
|------------|-------|---------|-------|
| Claude Desktop | `~/Library/Application Support/Claude/claude_desktop_config.json` | `%APPDATA%\Claude\claude_desktop_config.json` | `~/.config/Claude/claude_desktop_config.json` |
| Cursor | `~/.cursor/mcp.json` | `%APPDATA%\Cursor\mcp.json` | `~/.cursor/mcp.json` |
| Windsurf | `~/.codeium/windsurf/mcp_config.json` | `%APPDATA%\.codeium\windsurf\mcp_config.json` | `~/.codeium/windsurf/mcp_config.json` |

Определи OS пользователя (`uname -s` или `$env:OS`) и используй правильный путь.

При добавлении MCP — не перезаписывать существующие серверы с тем же именем, только добавлять новые.

## Обновление (повторный запуск)

Если bootstrap запускается повторно (уже установлено):
- Обновить только блок между `<!-- ai-setup:start -->` и `<!-- ai-setup:end -->`
- Добавить новые скиллы, не трогать существующие
- Добавить новые MCP, не трогать существующие
- Не пересоздавать файлы памяти
