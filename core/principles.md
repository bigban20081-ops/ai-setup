# Базовые принципы AI-разработки

Универсальные принципы для любого проекта и любого стека.
Bootstrap-агент генерирует стек-специфичные правила поверх этих.

---

## 1. Поведение агента

- Отвечать на языке пользователя
- Без эмодзи если явно не попросили
- Ответы краткие — без пересказа того, что только что было сделано
- Никогда не коммитить без явного запроса пользователя
- Если задача расплывчатая — задать 2–3 уточняющих вопроса до планирования, не после
- Не добавлять фичи, рефакторинг или "улучшения" сверх того, что просили
- Не добавлять обработку ошибок и валидацию для сценариев, которые не могут произойти
- Не создавать хелперы и абстракции для одноразового использования
- Не проектировать под гипотетические будущие требования

---

## 2. Workflow разработки

Для любой нетривиальной задачи — следовать этим шагам по порядку. Не пропускать, не переставлять.

**Шаг 1 — Уточнение**
Если задача расплывчатая — задать 2–3 вопроса до планирования. Не угадывать требования.

**Шаг 2 — План**
Составить список файлов, порядок изменений, риски, способ проверки.
Показать план и дождаться явного одобрения. Не приступать к коду без "ок, делай".

**Шаг 3 — Реализация**
Делать по плану. Если нужно отклониться — сначала сообщить и получить подтверждение.

**Шаг 4 — Проверка**
Три уровня, все обязательны:
- **Сборка** — запустить build/lint/typecheck проекта. Должно пройти без ошибок.
- **Функциональная** — реально запустить затронутый код. Для UI — открыть в браузере и проверить основной путь и edge cases. Если нельзя протестировать UI — сказать явно, не делать вид что проверил.
- **Безопасность** — для любого доступа к данным: проверить аутентификацию, фильтрацию по owner, отсутствие утечек между тенантами.

**Шаг 5 — Документация**
До коммита проверить, что нужно обновить:
- `CHANGELOG.md` — новая фича, фикс бага, изменение поведения, изменение биллинга, security
- Архитектурные доки — если изменилась структура модулей
- Файл инструкций (`CLAUDE.md` / `GEMINI.md` и т.д.) — если изменились критичные паттерны

**Шаг 6 — Коммит**
Одно логическое изменение на коммит. Формат — см. раздел Git.

### Когда можно сократить

Тривиальные задачи (исправить опечатку, переименовать переменную, добавить один импорт) не требуют полного цикла. При сомнениях — использовать полный цикл.

## 3. Git-конвенции

**Формат коммита:**
```
feat|fix|refactor|docs|chore: описание
```

**Правила:**
- Одно логическое изменение на коммит — не смешивать несвязанные правки
- Никогда не коммитить без явного запроса пользователя
- Перед коммитом убедиться что сборка проходит
- Перед стейджингом проверить `git status` — нет ли лишних файлов
- Перед удалением файла или экспорта — сначала grep на использования

```bash
grep -rn "import.*ComponentName\|from.*filename"
```

**Что идёт в CHANGELOG:**
- Новая фича, endpoint, страница, поле, интеграция
- Пользовательский баг-фикс
- Изменение поведения (пользователь увидит что что-то работает иначе)
- Миграция БД, влияющая на данные пользователей
- Изменение в биллинге
- Security-изменение с пользовательским эффектом

**Что НЕ идёт в CHANGELOG:**
- Внутренние рефакторинги без изменения поведения
- Изменения тестов без изменения кода
- Обновления документации
- Изменения CI/инструментария

## 4. Принципы кода

**Дисциплина скоупа:**
- Делать ровно то, что просили — не больше
- Никаких спекулятивных абстракций ("вдруг пригодится")
- Три похожих строки кода лучше преждевременной абстракции
- Не добавлять комментарии, docstring и аннотации типов в код, который не трогали
- Не комментировать то, что очевидно из кода

**Типизация:**
- Не дублировать интерфейсы — если тип используется в 2+ файлах, вынести в общее место
- Ответы API: создавать типизированные интерфейсы, не `Record<string, unknown>`
- Преобразование внешних форматов (snake_case и т.д.) — только на границе загрузки данных, не разбрасывать по коду
- Optional-поля: всегда null-safe доступ, никогда не предполагать наличие
- После изменения общих типов — запускать полную проверку типов: одно изменение может сломать 10+ файлов

**Удаление кода:**
- Перед удалением файла, экспорта или компонента — grep по всем использованиям
- Не доверять плану или предположениям — проверять поиском

**Логирование:**
- В контейнерных окружениях (Docker): только stdout/stderr, никаких файлов
- Не логировать чувствительные данные (токены, пароли, персональные данные)

## 5. Безопасность

**Аутентификация и авторизация:**
- Каждый endpoint/маршрут, обращающийся к данным пользователя, обязан требовать аутентификацию
- Никогда не отключать аутентификацию глобально — даже временно для отладки
- Дефолт при отсутствии аутентифицированного пользователя: пустой результат или 403, никогда все записи

**Изоляция данных (мультитенантность):**
- Каждый запрос к БД по данным пользователя должен фильтровать по owner/tenant
- Безопасный дефолт при отсутствии owner: вернуть пустой результат, не ошибку с утечкой данных
- Поля внешнего ключа для владения никогда не должны быть nullable
- Проверять владение и у связанных объектов, не только у основного ресурса

**Credentials и секреты:**
- API ключи, токены, пароли: всегда в переменных окружения, никогда в коде
- Никогда не коммитить `.env`-файлы и секреты в git
- Чувствительные данные в БД — шифровать
- Системные credentials (AI провайдеры, платёжные системы): только в `.env`
- Пользовательские credentials (OAuth токены, API ключи): шифровать в БД

**Токены сессии:**
- Auth-токены в httpOnly cookies, не в localStorage или sessionStorage
- OAuth callbacks: всегда проверять что возвращённый state принадлежит правильному owner

**Валидация:**
- Валидировать на системных границах (пользовательский ввод, внешние API, вебхуки)
- Не валидировать внутренние вызовы между доверенным кодом
- Санитизировать перед рендерингом пользовательского контента

**При отладке утечек данных:**
Проверять в порядке: ORM-фильтры → сериализаторы → авторизация на уровне view → ключи кэша.
Утечки данных почти всегда — отсутствующий фильтр, а не проблема кэширования.

## 6. Дизайн-система

**Контраст и читаемость:**
- Текст на цветном фоне: минимум 4.5:1 (WCAG AA)
- Бейджи и теги: насыщенный фон + белый текст — никогда светлый текст на светлом фоне
- Не передавать информацию только через цвет

**Консистентность:**
- Один дизайн-язык на проект: выбрать типографическую шкалу, систему отступов, border-radius — и придерживаться
- Отступы между секциями: консистентные кратные (например, базовая единица 8px)
- Hover-состояния на всех интерактивных элементах
- Одинаковый border-radius внутри семейства компонентов

**Адаптивность:**
- Mobile-first: проектировать с 375px, потом расширять
- Кнопки на мобильных — на всю ширину
- Горизонтальные сетки складываются вертикально на мобильных
- Touch targets минимум 44×44px
- Текст минимум 16px на мобильных (предотвращает zoom в iOS)

**Дисциплина компонентов:**
- Иконки и динамические импорты: только в клиентских компонентах (для SSR-фреймворков)
- Статические данные (для SEO): в серверных компонентах
- Не разделять компонент на server/client без реальной причины

**Анимации:**
- Цель: направить внимание, не развлекать
- Fade-in при скролле для секций (stagger 100–150ms между элементами)
- Hover: scale 1.02–1.05, 200ms ease
- Всегда уважать `prefers-reduced-motion`
- Не анимировать текст параграфов, навигацию, footer

## 7. SEO

**Каждая публичная страница обязана иметь:**
- `<title>` — макс. 60 символов, с основным ключевым словом
- `<meta name="description">` — макс. 155 символов, с призывом к действию
- Canonical URL
- Open Graph теги: `og:title`, `og:description`, `og:image` (1200×630px), `og:url`, `og:type`
- Twitter card теги

**Иерархия заголовков:**
- Один `<h1>` на страницу — основная тема
- `<h2>` для секций, `<h3>` для подразделов — не пропускать уровни
- Заголовки описывают контент, а не просто красиво выглядят

**Изображения:**
- Всегда описательный `alt` (не "image.png", не пустой для значимых изображений)
- Использовать оптимизированные компоненты фреймворка, не сырой `<img>` для контентных изображений
- Указывать `width` и `height` для предотвращения сдвига макета (CLS)
- Lazy load для всех изображений кроме hero/above-the-fold

**Structured data (Schema.org):**

| Тип страницы | Schema |
|---|---|
| Продукт/SaaS | `SoftwareApplication` + `Organization` + `FAQPage` |
| Услуги | `Service` + `Organization` + `FAQPage` |
| Блог | `Article` + `BreadcrumbList` |
| О компании | `Organization` + `Person` |
| Цены | `Offer` (только с реальными отзывами для `AggregateRating`) |

Правила:
- Один JSON-LD блок каждого типа — без дублей на странице
- Никогда не добавлять `aggregateRating` с выдуманными отзывами
- Хлебные крошки на всех страницах глубже корня

**Sitemap:**
- Обновлять при добавлении новых публичных страниц
- Исключать: страницы авторизации, дашборд, админ-панели, API маршруты

**Семантическая разметка:**
```
<header>   — навигация, один на страницу
<main>     — основной контент, один на страницу
<section>  — логические секции внутри main
<footer>   — подвал сайта
<nav>      — группы навигации
<article>  — самостоятельный контент (посты, карточки)
```

**Core Web Vitals (целевые значения):**
- LCP < 2.5s — оптимизировать hero-изображение, preload критических ресурсов
- CLS < 0.1 — размеры для img/video, `font-display: swap`
- INP < 200ms — минимум JS на первом экране

## 8. Дисциплина документации

Устаревшая документация — технический долг. Обновлять до коммита, не "потом".

**Триггеры обновления:**

| Что изменилось | Что обновить |
|---|---|
| Новая фича / endpoint / страница | CHANGELOG |
| Пользовательский баг-фикс | CHANGELOG |
| Изменилась структура модулей | Архитектурные доки |
| Изменился критичный паттерн или правило | Файл инструкций (CLAUDE.md / GEMINI.md и т.д.) |
| Security-инцидент | Лог инцидентов в файле инструкций: дата + что случилось + урок |
| Обнаружен неочевидный паттерн или грабля | Файл памяти проекта (insights, memory) |

**Запись инцидентов:**
При security-баге или утечке данных записать:
- Дата
- Что произошло
- Корневая причина
- Урок

Незаписанные баги повторяются. Запись занимает 5 минут и экономит часы.
